接口基础
接口

泛指实体把自己提供给外部的一种抽象化物（可以为另一个实体），用以由内部操作分离外部沟通方法；指系统或组件之间的交互点，数据交互的入口和出口，是一套规范、标准

接口优点：1、责任划分清晰；2、缩短研发周期、3、可拓展性强

API接口

应用程序编程接口（API:aapplication programming interface；以HTTP协议形式提供，定义了输入、输出、功能描述的服务

接口测试

主要是校验数据的交换、传递和控制管理过程；测试后端程序实现是否符合接口规范

为什么要做接口测试

1、系统复杂度不断上升，传统的测试方法成本增加效率下降；2、更容易是吸纳自动化持续集成，相对UI自动化比较稳定，减少人工回归测试的成本和时间；3、前后端架构是分离的，从安全层面，只是依赖前端进行限制已近完全不能满足系统的安全要求，需要后端同样进行控制；4、前后端传输、日志打印等信息是否加密传输也需要验证，特别设计到用户的隐私信息，如身份证、银行卡等

接口测试场景

阶段：功能测试前；针对接口文档整体做接口专项测试；性能测试时，通过接口请求进行并发；APP弱网测试，通过接口请求进行抓包

接口测试流程

1、需求分析：主要依据需求文档；2、接口文档解析：一般由开发人员编写接口文档（API文档），读懂接口清单,RUR+请求方法，提交的数据，响应的结果；3、设计测试用例；4、执行用例，接口测试工具实现；5、接口缺陷管理与跟踪；6、接口自动化持续集成

确认接口文档

接口文档基本信息：接口名称–确认接口的功能；接口地址–URL-确认请求地址准确性；请求参数支持格式–采用xml/json格式进行请求；请求方式–GET/POST/PUT/DELETE;请求参数（参数名称、类型、是否必填、参数说明等）–用例设计，覆盖所有参数，正向用例；返回参数（返回码、返回信息、返回json消息或xml消息）–预期结果

接口测试原理

模拟客户端向服务端发送请求，服务器接收到请求后进行的业务处理，冰箱客户端返回响应数据，检查响应数据是否符合预期结果（响应的结果、响应码、响应时间）

1、定位服务器资源（如：url+get）；2、模拟用户提交数据（url？键=值&键=值）；3、查看响应的结果是否符合预期（200+一条记录或多条记录）

特点：1、测试可提前介入，提早发现bug，符合质量控制前移动的理念；2、可发现一些页面操作发现不了的问题；3、低成本高效益（底层的一个bug能够引发上层多个bug，接口测试可以实现自动化持续集成）；4、不同于传统的单元测试，接口测试是从用户对系统进行全面的检测

接口规范Restful风格

RESTful（Representational State Transfer）表现层状态转换

JSON传递效率高，部分场景下使用JSON替换html和xml
但JSON语法描述性不及标签语言，部分场景还需使用html和xml
如果传递的是少量数据的话，可使用JSON

接口测试用例设计 测试要点

通用信息校验：
url校验：1、正确url，2、错误url
请求方法校验：
1、对应正确的请求方法，如发生get请求查询信息，
2、错误的请求方法
请求头：1、正确传递请求头，如Content-Type值：application/json,2、错误的传递请求头
接口鉴权：token或session、 1、正确认证信息，2、错误认证信息、失效的信息、
3、不传认证信息

接口参数校验：
1、参数的必填项校验：参数正确、不传、为空、空字符串
2、参数的选填项校验：选填、不填
3、参数长度校验：正常长度、大于、小于、等于
4、参数数据类型校验：正确数据类型、错误数据类型
5、参数的有效性校验：有范围内，无效范围内
6、参数的唯一性校验：唯一字段数据唯一、唯一字段数据不唯一，不唯一字段不唯一
7、参数关联项校验：参数存在关联

其他补充项：幂等性：重复提交和一次提交的效果是相同的，如 抽奖
弱网环境：事务的完备性，付款过程中接口数据发送时网络中断扣款收款是否成功
服务架构：分布式系统，能否正确同步数据，Nginx配置模拟
接口风格：检查是否符合RESTful风格
敏感信息加密：如登录接口用户信息用密文传输

接口请求信息中，重点关注4大信息
接口URL地址、请求方法、请求头、请求参数

接口测试三要素
请求地址、请求参数、返回数据
